# Верхнеуровневый план реализации защищённого API для LLM

**Версия:** согласована с `ТЗ_0.txt`, `Критерии_оценки_fastapi_llm.txt` и уточнениями ниже.

## 0. Структура каталогов и точка входа (сведение с ТЗ)

В **ТЗ** задан запуск вида:

`uv run uvicorn app.main:app --reload ...`

и дерево модулей с пакетом **`app/`** на верхнем уровне проекта приложения (`app/core`, `app/db`, …). Критерии оценивают наличие каталогов **`app/`, `core/`, `db/`** и т.д. именно как часть приложения.

**Требование к реализации:** расположить код так, чтобы импорт `app.main` и команда из ТЗ выполнялись **без изменения семантики** (либо код в корне репозитория под `app/`, либо явно описать в `README.md` альтернативную команду `uvicorn`, эквивалентную по смыслу, и поддерживать её в установке через `uv`). Цель — чтобы проверяющий воспроизвёл окружение по документации без ошибок импорта.

---

## 1. Архитектурная концепция

**Многослойная архитектура с чёткими границами ответственности:**

- **HTTP слой (API)** — только маршрутизация, валидация входных данных, преобразование доменных ошибок в HTTP-ответы
- **UseCase слой** — бизнес-логика, оркестрация репозиториев и сервисов
- **Repository слой** — атомарные операции с БД (только SQL/ORM)
- **Service слой** — внешние интеграции (OpenRouter)

---

## 2. Технологический стек

- **Фреймворк:** FastAPI (асинхронный)
- **База данных:** SQLite + aiosqlite (асинхронный драйвер)
- **ORM:** SQLAlchemy 2.0 (асинхронный)
- **Аутентификация:** JWT (`python-jose`) + bcrypt (`passlib`)
- **HTTP клиент:** `httpx` (асинхронный)
- **Управление зависимостями:** `uv`
- **Линтер:** `ruff`

Зависимости и версии — по списку из **ТЗ** в `pyproject.toml` (`fastapi`, `uvicorn`, `sqlalchemy`, `aiosqlite`, `pydantic-settings`, `python-jose[cryptography]`, `passlib[bcrypt]`, `httpx`, `python-multipart`, `greenlet`, зафиксированный `bcrypt`, `ruff`).

---

## 3. Ключевые компоненты системы

### 3.1. Инфраструктурный слой

- Конфигурация через **Pydantic Settings** (чтение `.env`): в т.ч. `APP_NAME`, `ENV`, JWT, `SQLITE_PATH`, параметры OpenRouter (`OPENROUTER_API_KEY`, `OPENROUTER_BASE_URL`, модель, `OPENROUTER_SITE_URL`, `OPENROUTER_APP_NAME` для заголовков)
- Асинхронное подключение к SQLite (`sqlite+aiosqlite:///...`)
- JWT утилиты (генерация/верификация) — см. §4.1
- Хеширование паролей — в **`core/security`**, без обращений к БД из этого модуля
- **Доменные исключения** — см. §4.3 (включая тип «запрещено», если появляются проверки прав)

### 3.2. Слой данных (SQLAlchemy)

- Модель **`User`**: `id`, `email`, `password_hash`, `role`, `created_at`
- Модель **`ChatMessage`**: `id`, `user_id`, `role`, `content`, `created_at`
- Связь **one-to-many** между `User` и `ChatMessage`

### 3.3. Репозитории

- **`UserRepository`**: создание пользователя, поиск по email/id (`commit`/`refresh` на уровне репозитория по ТЗ)
- **`ChatMessageRepository`**: добавление сообщения; выборка **последних N** с нужной сортировкой; **полная очистка** истории пользователя  
  Формирование списка `messages` для LLM остаётся в **ChatUseCase**, не в репозитории.

### 3.4. Бизнес-логика (UseCases)

- **`AuthUseCase`**: регистрация (уникальность email → доменный конфликт; хеширование в usecase или через `security`), логин (верификация пароля → JWT с полным payload по §4.1), профиль по `user_id` (не найден → доменная ошибка)
- **`ChatUseCase`**: сбор контекста (опционально `system` → история из репозитория → текущий `user` prompt); сохранение пары сообщений пользователь/ассистент; параметры **`max_history`**, **`temperature`**; история только текущего пользователя

### 3.5. Внешний сервис

- **`OpenRouterClient`**: `POST …/chat/completions`; заголовки **`Authorization`**, **`HTTP-Referer`**, **`X-Title`** из настроек; тело — модель из конфигурации и массив `messages` по роли/контенту; при ошибке HTTP/API — доменная **ошибка внешнего сервиса** без «падения» процесса на необработанное исключение

**Модель по умолчанию (критерии приёмки и пример `.env` в ТЗ):**  
`stepfun/step-3.5-flash:free` задаётся через настройки (переменная уровня `OPENROUTER_MODEL` или эквивалент в вашем `Settings`).

### 3.6. API слой (эндпоинты)

- **Auth:** `POST /auth/register`, `POST /auth/login` (OAuth2 form, `username` = email), `GET /auth/me`
- **Chat:** `POST /chat`, `GET /chat/history`, `DELETE /chat/history`
- **`GET /health`** — статус и окружение (по ТЗ для `main` / сборки приложения)
- По ТЗ допускается **CORS** при наличии флагов в конфиге — при необходимости заложить в `create_app()`

## 4. Ключевые технические решения

### 4.1. Аутентификация

- **`OAuth2PasswordBearer`** для интеграции со Swagger UI (`Authorize`)
- Логин — **`OAuth2PasswordRequestForm`**
- **JWT payload (обязательно по ТЗ):** **`sub`** (идентификатор пользователя, `user_id`), **`role`**, **`exp`**, **`iat`**  
  В концепции ранее фигурировали только `sub`, `exp`, `iat`; для соответствия ТЗ **`role` включается в каждый access token**.
- Dependency **`get_current_user_id`** (или эквивалент): декодирование → извлечение `sub`; доменные/HTTP-ошибки 401 по правилам ТЗ для API-слоя / deps

### 4.2. Управление контекстом LLM

- Опциональная **system**-инструкция
- Лимит истории — **`max_history`**
- **`temperature`** в запросе к провайдеру

### 4.3. Обработка ошибок

Исключения домена (без **`HTTPException` внутри usecase**), с маппингом в роутере / глобальными обработчиками:

| Смысл (ТЗ) | Пример в концепции |
|------------|---------------------|
| Конфликт (email занят и т.п.) | `ConflictError` |
| Не авторизован (неверный пароль и т.п.) | `UnauthorizedError` |
| **Запрещено (нет прав)** | **`ForbiddenError`** (или эквивалент; использовать при появлении проверок прав по `role`; иначе класс описать и зарезервировать маппинг в HTTP 403) |
| Не найдено | `NotFoundError` |
| Внешний сервис (OpenRouter) | `ExternalServiceError` |

Базовый класс приложения для всех доменных ошибок — по ТЗ (`app/core/errors.py`).

### 4.4. Инициализация приложения

- Фабричная функция **`create_app()`**
- На **startup**: создание таблиц через **`Base.metadata.create_all`** (асинхронный engine по ТЗ)
- Подключение роутеров auth и chat, без бизнес-логики в `main`

## 7. Самопроверка при реализации и перед сдачей

Практическая выжимка из рецензии: три опоры — **границы слоёв**, **контролируемые сбои OpenRouter**, при желании — **минимальные тесты** (не требуются критериями напрямую, снижают регресс).

### 7.1. Чеклист разделения слоёв

- **`api/routes_*.py`** — только `Depends`, Pydantic-схемы, вызовы use case, перевод доменных ошибок в HTTP (или единые exception handlers). **Нет:** прямых `AsyncSession`/репозиториев, SQL, клиента `httpx`, логики OpenRouter и бизнес-правил не из use case.

- **`usecases/*`** — сценарии, вызовы репозиториев и `OpenRouterClient`, выброс **только доменных** исключений. **Нет:** `HTTPException`, прямых SQL/Session.

- **`repositories/*`** — только ORM и операции с данными. **Нет:** хеширования паролей, JWT, вызовов OpenRouter, сборки списка `messages` для LLM (контекст — задача **`ChatUseCase`**).

- **`services/openrouter_client.py`** — HTTP к OpenRouter, заголовки и тело по ТЗ, при ошибке — **одна** понятная ветка в доменное **`ExternalServiceError`** (или эквивалент). **Нет:** доступа к БД и типов FastAPI.

- **`core/security`** — хеши и JWT (**включая `role` в payload** по §4.1). **Нет:** обращений к базе.

- **`api/deps.py`** — сессии, сборка репозиториев/use case, **`get_current_user_id`** из токена. По ТЗ здесь допустим перевод части ошибок аутентификации в **HTTP 401**.

### 7.2. Ошибки OpenRouter (минимум для критерия «не падаем»)

- Ответ провайдера **4xx/5xx** → не голый traceback: доменное **`ExternalServiceError`** (сообщение/детали для лога при необходимости).

- **Таймаут / сеть** (`httpx` и т.п.) → та же доменная ошибка, не проброс необработанного клиента наверх API.

- **HTTP 200, но отсутствует ожидаемое тело ответа** (нет choices/content) → ошибка провайдера, снова доменное исключение.

- В API-слое — **единый стиль** превращения этого исключения в HTTP (**например 502 или 503**; главное — предсказуемо для клиента и **без 500 при ожидаемых сбоях внешнего API**).

### 7.3. Опциональные тесты (если остаётся время)

Не входят в обязательный минимум критериев приёмки; полезно перед защитой от регресса:

- **Auth use case:** дубликат email → конфликт; неверный пароль / несуществующий пользователь → неавторизован.

- **Chat use case:** мок **`OpenRouterClient`** — успех → два сохранения в репозиторий (сообщение пользователя и ответ assistant) в ожидаемом порядке; клиент кидает **ExternalServiceError** → зафиксировать согласованное поведение (что сохранено в истории до/после сбоя — как заложено в реализации).

- **Минимальная интеграция:** `TestClient` + тестовая БД / подменённые зависимости, без реального вызова OpenRouter.

Рекомендуемый момент: после стабилизации сценариев из этапа **7** плана реализации (§5).
