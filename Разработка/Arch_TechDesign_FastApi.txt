# Архитектурная спецификация проекта (Технический дизайн)

**Проект:** Защищённый API для взаимодействия с LLM через OpenRouter  
**Версия:** 1.1  
**Статус документа:** архитектурная и техническая спецификация **утверждена** для реализации. **Готовность продукта к сдаче** определяется закрытием чеклистов (в первую очередь эпик 7), а не этой строкой статуса.

Документ описывает целевую архитектуру серверного приложения на FastAPI, предоставляющего защищённый API для взаимодействия с большой языковой моделью (LLM) через сервис OpenRouter, и синхронизирован с `project/ТЗ_0.txt` и `project/Критерии_оценки_fastapi_llm.txt`.

**Про чеклисты в эпиках:** в задачах ниже пункты имеют вид `- [ ]` — **реквизиты готовности**; по мере реализации в коде отмечайте выполнение `- [x]`, чтобы документ отражал фактическое состояние репозитория.

## Согласованные архитектурные решения

| Пункт ТЗ / черновика | Утверждённая реализация | Обоснование |
|----------------------|-------------------------|-------------|
| JWT payload содержит только `sub`, `exp`, `iat` | **Полный payload:** `sub`, `role`, `exp`, `iat` | ТЗ в разделе `core/security.py` явно требует поле `role` в токене |
| Доменные ошибки без `ForbiddenError` | **Включена `ForbiddenError`** для соответствия ТЗ (даже если в MVP не используется) | ТЗ перечисляет "запрещено (нет прав)" среди доменных исключений |
| Структура корня не специфицирована | **Пакет `app/` внутри каталога `llm_p/`**; запуск **`uvicorn app.main:app` из `llm_p/`** | Критерии оценивают слои `app/`, `core/`, `db/` и т.д.; рабочий каталог — см. `llm_p/README.md`. |
| Модель OpenRouter не зафиксирована | **`stepfun/step-3.5-flash:free`** как значение по умолчанию | Критерии оценки явно проверяют использование этой модели |

## Общая архитектура

### 1. Контракты между компонентами

#### Контракт API слой → UseCase слой

```python
# Интерфейс UseCase (неявный, через Dependency Injection)
interface AuthUseCase:
    async def register(email: str, password: str) -> UserPublic
    async def login(email: str, password: str) -> TokenResponse
    async def get_profile(user_id: int) -> UserPublic

interface ChatUseCase:
    async def ask(
        user_id: int,
        prompt: str,
        system: str | None,
        max_history: int,
        temperature: float
    ) -> ChatResponse
    async def get_history(user_id: int, limit: int | None) -> list[ChatMessage]
    async def clear_history(user_id: int) -> None
```

**Обязательные правила:**

- API слой **не вызывает репозитории напрямую**.
- API слой **не содержит SQLAlchemy-кода**.
- UseCase слой **не выбрасывает HTTPException** — только доменные исключения.
- UseCase слой **не зависит от FastAPI** (нет импортов `fastapi`, `Request`, `Depends`).

#### Контракт UseCase слой → Repository слой

```python
# Интерфейс UserRepository
interface UserRepository:
    async def get_by_email(email: str) -> User | None
    async def get_by_id(user_id: int) -> User | None
    async def create(email: str, password_hash: str, role: str) -> User

# Интерфейс ChatMessageRepository
interface ChatMessageRepository:
    async def add_message(user_id: int, role: str, content: str) -> ChatMessage
    async def get_last_n(user_id: int, n: int) -> list[ChatMessage]
    async def delete_all(user_id: int) -> None
```

**Обязательные правила:**

- Репозитории **не хешируют пароли**.
- Репозитории **не вызывают OpenRouter**.
- Репозитории **не содержат бизнес-логику** (только CRUD/ORM).
- Репозитории **не импортируют `fastapi` и не выбрасывают HTTPException**.

#### Контракт UseCase слой → Service слой

```python
# Интерфейс OpenRouterClient
interface OpenRouterClient:
    async def chat_completion(
        messages: list[dict[str, str]],
        temperature: float
    ) -> str
```

**Обязательные правила:**

- Сервис **не обращается к базе данных**.
- Сервис **не знает о пользователях** (не принимает `user_id`).
- При ошибках внешнего API сервис выбрасывает **`ExternalServiceError`**.

#### Контракт JWT (Auth Service → всё приложение)

```python
# JWT Payload interface
interface JWTToken:
    sub: str          # user_id (строка, для совместимости)
    role: str         # "user" | "admin"
    iat: int          # issued at, Unix timestamp
    exp: int          # expires at, Unix timestamp
```

**Обязательные правила:**

- JWT выпускается только **при успешном логине** (use case).
- Валидация JWT происходит **в слое dependency (`deps.py`)**.
- При невалидном токене dependency выбрасывает **`HTTPException(401)`**.
- При декодировании проверяются: подпись, `exp`, наличие `sub` и `role`.

## 3. Схема данных

| Компонент | Хранилище | Данные | TTL | Ключ / Таблица |
|-----------|-----------|--------|-----|----------------|
| Приложение | SQLite (aiosqlite) | users: id, email, password_hash, role, created_at | бессрочно | таблица `users` |
| Приложение | SQLite (aiosqlite) | chat_messages: id, user_id, role, content, created_at | бессрочно | таблица `chat_messages` |
| Приложение | Файловая система | база данных `app.db` | бессрочно | `./app.db` (из настроек) |

**Связи:**
- `chat_messages.user_id` → `users.id` (внешний ключ; в ORM задать **`ondelete="CASCADE"`** у FK, чтобы при удалении пользователя удалялись его сообщения — согласовано с каскадом в спецификации)

**Индексы:**
- `users.email` — уникальный индекс
- `chat_messages.user_id`, `chat_messages.created_at` — для быстрой выборки истории

## Эпик 3: Безопасность и утилиты

### Цель

Реализовать криптографические утилиты (JWT, bcrypt) и доменные исключения.

### Технический дизайн

1. **JWT:** `python-jose` с алгоритмом `HS256`.
2. **Хеширование:** `passlib[bcrypt]` с `bcrypt=4.3.0`.
3. **Ошибки:** Иерархия доменных исключений в `core/errors.py`.

### Задача 3.1: Хеширование паролей

**Acceptance Criteria:**
- [ ] `hash_password(password: str) -> str` (passlib)
- [ ] `verify_password(plain: str, hashed: str) -> bool`

**Файлы:**
- `app/core/security.py`

### Задача 3.2: JWT функции

**Acceptance Criteria:**
- [ ] `create_access_token(data: dict) -> str`
- [ ] Payload содержит: `sub` (user_id как str), `role`, `exp`, `iat`
- [ ] `decode_token(token: str) -> dict` (валидация подписи и `exp`)
- [ ] Используются `JWT_SECRET` и `JWT_ALG` из настроек

**Файлы:**
- `app/core/security.py`

### Задача 3.3: Доменные исключения

**Acceptance Criteria:**
- [ ] Базовый класс `AppError` (наследник `Exception`)
- [ ] `ConflictError` — 409 (email уже существует)

- [ ] `ForbiddenError` — 403 (нет прав, задел на будущее)
- [ ] `NotFoundError` — 404 (пользователь/сообщение не найдены)
- [ ] `ExternalServiceError` — 502/503 (ошибка OpenRouter)

**Файлы:**
- `app/core/errors.py`

## Эпик 5: Бизнес-логика (UseCases)

### Цель

Реализовать сценарии регистрации, логина и работы с чатом.

### Технический дизайн

1. **AuthUseCase:** регистрация (проверка уникальности, хеширование), логин (верификация + JWT).
2. **ChatUseCase:** сбор контекста, вызов LLM, сохранение истории.

### Задача 5.1: AuthUseCase

**Acceptance Criteria:**
- [ ] `register(email, password) -> UserPublic`
  - Проверка: email не занят → иначе `ConflictError`
  - Хеширование пароля, создание пользователя с role="user"
- [ ] `login(email, password) -> TokenResponse`
  - Если пользователь **не найден** или пароль **не совпадает** → **одинаково** `UnauthorizedError` (не раскрывать факт существования email; не отдавать `NotFoundError` на логин)
  - При успехе: генерация JWT с `sub=str(user.id)`, `role=user.role`
- [ ] `get_profile(user_id) -> UserPublic`
  - Поиск пользователя → `NotFoundError`

**Файлы:**
- `app/usecases/auth.py`

### Задача 5.2: ChatUseCase

**Acceptance Criteria:**
- [ ] `ask(user_id, prompt, system, max_history, temperature) -> ChatResponse`
  - Получить историю (до `max_history` сообщений)
  - Собрать `messages`: опционально system → история → текущий prompt
  - Сохранить prompt в БД (role="user")
  - Вызвать `openrouter_client.chat_completion(messages, temperature)`
  - Сохранить ответ в БД (role="assistant")
  - Вернуть ответ
- [ ] `get_history(user_id, limit) -> list[ChatMessage]`
  - Вызов репозитория с limit (по умолчанию 50)
- [ ] `clear_history(user_id) -> None`
  - Вызов `chat_message_repository.delete_all(user_id)`

**Файлы:**
- `app/usecases/chat.py`

## Эпик 6: HTTP слой (API endpoints, DI)

## Итоговое архитектурное решение

**Целевая архитектура слоёв и интерфейсы между ними утверждены как техническая спецификация.** Фактическая готовность кода и формальная приёмка — по выполнению чеклистов эпиков (особенно эпика 7).

**Обязательные архитектурные инварианты:**

1. **Слои изолированы:** API → UseCases → Repositories → DB / Services
2. **Эндпоинты не содержат бизнес-логики** — только вызовы usecase
3. **Репозитории не знают о JWT, хешировании и OpenRouter**
4. **UseCase не выбрасывают `HTTPException`** — только доменные исключения; **AuthUseCase** при логине не различает для клиента «email не найден» и «неверный пароль» — **`UnauthorizedError`** в обоих случаях
5. **JWT содержит `sub`, `role`, `exp`, `iat`** (role обязателен)
6. **OpenRouter использует модель `stepfun/step-3.5-flash:free`**
7. **База данных — SQLite + aiosqlite**; DDL при старте только **async-safe** способом (см. эпик 6.4), без синхронного **`create_all(bind=async_engine)`**
8. **Зависимости управляются через `uv`**, проект воспроизводим в чистом окружении
9. **README содержит все скриншоты** с email формата `student_surname@email.com`
10. **`ruff check` проходит без ошибок**
