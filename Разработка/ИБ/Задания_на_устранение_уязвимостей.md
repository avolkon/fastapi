# Задания на разработку: устранение выявленных рисков ИБ

Все задачи ниже связаны с ревью в файле `Ревью_безопасности_кодовая_база.md` в этой же папке. Порядок выполнения можно менять по приоритету продукта, но пункты помечены **P0/P1/P2/P3** для ориентира.

---

## Задание 1 (P0). Политика силы `JWT_SECRET` и прод-конфигурация JWT

**Проблема.** Секрет может быть случайно слабым или коротким приложение принимает любую строку из окружения.

**Цель.**

- При старте в окружении `ENV=production` или аналоге отклонять слишком короткий или пустой `JWT_SECRET` с явной ошибкой запуска.
- Задокументировать в README и в `.env.example` требование длины и энтропии например минимум 32 байта в случайном виде только пример генерации через `secrets.token_urlsafe(...)`.

**Критерий приёмки.** Некорректный секрет в проде приложение не поднимает в дев режиме поведение согласуется с политикой команды допуск может быть смягчён с предупреждением только при явном флаге.

## Задание 2 (P0). Allowlist базового хоста OpenRouter

**Проблема.** Произвольный `OPENROUTER_BASE_URL` может направить исходящий HTTP-трафик на недоверенный хост при компромиссе конфигурации.

**Цель.**

- Ввести в настройках флаг например `LLM_PROVIDER_URL_STRICT=true` или жёсткую переменную `OPENROUTER_BASE_URL_ALLOWED_HOSTS`.
- При инициализации `OpenRouterClient` проверять, что распарсенный hostname входит в allowlist например только `openrouter.ai` или подконтрольные прокси по согласованию.
- В продакшене по умолчанию включать строгий режим.

**Критерий приёмки.** Недопустимый хост при строгом режиме приводит к ошибке при старте или к контролируемому `ExternalServiceError` без исходящего запроса на произвольный URL.

## Задание 3 (P1). Механизм отзыва или срока жизни access-токенов

**Проблема.** Украденный JWT действителен до `exp`.

**Цель.**

- Вариант A: ввести refresh-токен с коротким access и более длинным refresh с хранением refresh в HttpOnly cookie или только на сервере по политики проекта как в методичке.
- Вариант B: блоклист JWT по `jti` в БД или кэше с проверкой на каждый запрос и эндпоинт `POST /auth/logout`.
- Вариант C минимально: только существенное сокращение `ACCESS_TOKEN_EXPIRE_MINUTES` в проде плюс ротация `JWT_SECRET` по процессу эксплуатации до внедрения A или B считается временной мерой описать ограничения в документации.

**Критерий приёмки.** Описано в решении архитектуры и есть тест или ручная инструкция подтверждающая отзыв после выхода из системы или смены секрета.

## Задание 4 (P1). Ограничение частоты запросов (rate limiting)

**Проблема.** Нет защиты от брутфорса и массовых вызовов.

**Цель.**

- Ввести rate limit на `POST /auth/login` и `POST /auth/register` по ключу комбинация IP и или email нормализованный с разумными лимитами и ответом 429.
- Ввести ограничение на `POST /chat` для аутентифицированного пользователя по user_id чтобы ограничить злоупотребление ключом провайдера.

**Реализация.**

- Middleware на базе памяти для одного узла или Redis при горизонтальном масштабировании готовые пакеты по согласованию архитектуры допустимы.

**Критерий приёмки.** При превышении порога код 429 с нейтральным телом сообщения документация обновлена.

---

## Задание 5 (P1). Лимиты размера контекста запроса к LLM

**Проблема.** Большое `max_history` и длинные поля умножают объём запроса к провайдеру.

**Цель.**

- Снизить верхнюю границу `max_history` например до 50–100 либо ввести второй предел суммарной длины символов или оценки токенов для списка `messages` перед вызовом `chat_completion`.
- При превышении отвечать 400 с понятным `detail` без вызова внешнего API.

**Критерий приёмки.** Граничный тест подтверждает отсечение до вызова OpenRouter.

---

## Задание 6 (P2). Политика роли в JWT и будущие эндпоинты

**Проблема.** Поле `role` в токене не сверяется с БД при текущих маршрутах.

**Цель.**

- Зафиксировать в архитектурной заметке правило при появлении админских операций роль и права читать из БД или из кэша подписанного снимка пользователя а не доверять только JWT.
- Опционально удалить `role` из payload если не используется либо добавить валидатор при выдаче токена что роль совпадает с записью в БД.

**Критерий приёмки.** Документ и при необходимости код не оставляют двусмысленности для следующих задач авторизации.

## Задание 7 (P2). Обезличить или ограничить `/health`

**Проблема.** Поле `env` раскрывает тип среды без аутентификации.

**Цель.**

- В проде возвращать только `{"status": "ok"}` или использовать отдельный внутренний эндпоинт `/internal/health` за сетью доверия если нужны детали.
- Альтернативно включать детали только при `ENV=local`.

**Критерий приёмки.** Обновление README поведение согласовано с эксплуатацией.

---

## Задание 8 (P2). Политика ответов при конфликте регистрации

**Проблема.** Регистрация может раскрывать наличие email.

**Цель.**

- Вариант A: унифицировать ответ успех vs конфликт для внешнего клиента с одинаковым HTTP-кодом и телом там где методичка позволяет не всегда корректно при REST.
- Вариант B оставить 409 но нейтральный текст без упоминания email и включить описание допустимого trade-off информативность против приватность в методичке.
- Вариант C: rate limit на регистрацию см задание 4.

**Критерий приёмки.** Согласованный список текстов ошибок документирован решение утверждено владельцем продукта.

---

## Задание 9 (P2). Ужесточить конфигурацию CORS под сценарии

**Проблема.** `allow_methods` и `allow_headers` универсальные.

**Цель.**

- Ограничить методы и заголовки до реально нужных для Swagger и браузера фронта.
- Пересмотреть необходимость `allow_credentials` при чистой схеме Bearer в заголовке.

**Критерий приёмки.** Прогон браузерного клиента против стенда завершён ошибок CORS нет там где ожидалась работа с API.

## Задание 10 (P2). Интегритет ролей в истории сообщений

**Проблема.** Для LLM собираются `role`/`content` из БД при аномальных данных возможны нежелательные эффекты на промпт.

**Цель.**

- При чтении истории фильтровать или нормализовать допустимый набор `role` перед отправкой провайдеру например только `user` и `assistant` для блока истории `system` отдельно из запроса.
- Рассмотреть ограничение на уровне БД ENUM или проверку при записи сообщений только из доверенных путей.

**Критерий приёмки.** Ручной или автоматический сценарий с подменённой записью если допускается в тестах не должно быть неучтённых ролей в теле outbound запроса.

## Задание 11 (P2). Контракт текстов ошибок домена

**Проблема.** Клиенту отдаётся `exc.message` целиком.

**Цель.**

- Ввести для фасада API словарь кода ошибки публичный текст и необязательный внутренний detail в логи без отдачи клиенту.
- Применить хотя бы к классам `ExternalServiceError` и к прочим ошибкам с потенциально длинными сообщениями.

**Критерий приёмки.** Интеграционный тест подтверждает что в теле ответа нет внутренних URL полных трасс необработанное исключение по-прежнему не утекает в production middleware.
