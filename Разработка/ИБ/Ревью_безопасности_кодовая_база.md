# Ревью безопасности кодовой базы (FastAPI / app)

**Объект:** репозиторий `fastapi`, слой приложения `app`.  
**Предпосылки:** точка зрения архитектора ИБ и white box; без гарантий полноты вне этого дерева и без аудита конкретного хостинга (TLS, firewall, SELinux задаются эксплуатацией).

Дата подготовки документа: 2026-05-11.

## 1. Область проверки

Просмотрены в основном: `core/security.py`, `core/config.py`, `core/errors.py`, `api/deps.py`, `api/routes_auth.py`, `api/routes_chat.py`, `main.py`, `services/openrouter_client.py`, `usecases/auth.py`, `usecases/chat.py`, `schemas/auth.py`, `schemas/chat.py`, а также упомянутые в обзоре зависимости и конфигурация через переменные окружения.

## 2. Критические риски (конфигурация и эксплуатация)

### 2.1. Секрет JWT и параметры алгоритма из окружения

**Суть.** `JWT_SECRET` и `JWT_ALG` задаются через настройки. Слабый или короткий секрет равносилен возможности выпускать валидные токены. Неверная эксплуатация алгоритмов в связке с версиями криптобиблиотек традиционно относится к классу ошибок конфигурации и историческим проблемам библиотек (в коде указан явный список `algorithms` при декодировании, что снижает один из векторов, но не отменяет требование к качеству секрета).

**Фрагмент кода.**

```python
# app/core/security.py — выпуск и разбор JWT
return jwt.encode(
    payload,
    settings.jwt_secret,
    algorithm=settings.jwt_alg,
)
...
return jwt.decode(
    token,
    settings.jwt_secret,
    algorithms=[settings.jwt_alg],
)
```

---

### 2.2. Произвольный `OPENROUTER_BASE_URL` (SSRF с позиции сервера приложения)

**Суть.** Базовый URL провайдера LLM задаётся в конфигурации; клиент выполняет `POST` на `{base}/chat/completions`. Пользователь API этот параметр напрямую не передаёт, но при компромиссе секретов окружения или ошибочной настройке возможен запрос ко внутренним адресам из доверенной сети процесса (метаданные облаков, localhost, смежные сервисы).

**Фрагмент кода.**

```python
# app/services/openrouter_client.py
response = await client.post(
    f"{self._base}/chat/completions",
    ...
)
```

---

### 2.3. Отсутствие отзыва access-токенов

**Суть.** Выдаётся JWT с полем `exp`. При компромиссе одного токена он действует до истечения срока. Нет механизма denylist/blacklist нет связки с refresh-сессией с ротацией в коде приложения не реализована.

## 3. Высокие риски

### 3.1. Нет ограничения частоты запросов (rate limiting)

**Суть.** Эндпоинты вида `/auth/login`, `/auth/register`, `POST /chat` не имеют ограничений частоты на уровне приложения. Это открывает путь к брутфорсу (хотя логин и не даёт явного разделения кейсов «нет пользователя» и «не тот пароль»), к спам-регистрациям и к злоупотреблению исходящими вызовами к провайдеру LLM со стороны авторизованного пользователя или при компромиссе множества учётных записей.

---

### 3.2. Кумулятивный размер запроса к LLM и злоупотребление контекстом

**Суть.** Одно поле промпта ограничено схемой, но параметр `max_history` может достигать 500 сохранённых сообщений из БД. В совокупности с `system` и текущим `prompt` запрос к внешнему API становится большим по объёму, что создаёт риск расхода квот, задержек и косвенного отказа в обслуживании по исходящему каналу.

**Фрагмент схемы.**

```python
# app/schemas/chat.py
max_history: int = Field(default=16, ge=0, le=500)
```

---

### 3.3. Роль в JWT не используется для авторизации в защищённых маршрутах

**Суть.** В токен кладётся `role`, но зависимость `get_current_user_id` использует только `sub`. При появлении в будущем эндпоинтов только с проверкой роли из токена без сверки с БД возможна ошибка архитектуры (доверие к полю токена при утере секретного ключа становится особенно критичной).

---

### 3.4. Утечка информации через `/health`

**Суть.** Ответ включает `env` из настроек приложения (например, `local` против продакшен-метки). Это облегчает профилирование среды наблюдающим без аутентификации.

**Фрагмент кода.**

```python
# app/main.py
return {"status": "ok", "env": settings.env}
```

## 4. Средние риски

### 4.1. Перечисление пользователей при регистрации

**Суть.** Сообщение о том, что email уже занят (конфликт при регистрации), намеренно отличает занятость адреса от сценария логина, где ошибка унифицирована. Это ослабление приватности идентификаторов (user enumeration для регистрации).

---

### 4.2. Конфигурация CORS с `credentials` и широкими методами или заголовками

**Суть.** При непустом списке origin включаются `allow_credentials=True`, `allow_methods=["*"]`, `allow_headers=["*"]`. При ошибке в списке `CORS_ORIGINS` (слишком широкое значение origin) возможны проблемы в сценариях с cookie и браузерами совместно с Bearer обычно риск ниже, но конфигурация остаётся чувствительной.

---

### 4.3. Доверие к полям `role`/`content` в истории сообщений для LLM

**Суть.** Контекст для модели собирается из записей БД парами `role` и `content`. При нарушении целостности данных или ошибках записи возможны нестандартные роли для модели класс промпт-модификации на уровне LLM не HTTP-XSS приложения напрямую.

---

### 4.4. Зависимости с криптографией (`python-jose`, `passlib`, `bcrypt`)

**Суть.** У криптографических цепочек периодически появляются рекомендации по версиям. Без закреплённых версий в производственной среде и мониторинга advisories возможен незамеченный заход регрессий или известных дыр в цепочке поставки.

---

### 4.5. Вывод текстов доменных ошибок клиенту

**Суть.** Обработчик `AppError` отдаёт `exc.message`. Если когда-либо туда попадёт внутренняя деталь приложения возможна вторичная утечка данных в отличие от заранее оговорённых текстов ошибок это требует дисциплины разработчиков при создании исключений.

---

## 5. Низкие и информационные наблюдения

- **Политика паролей.** Минимальная длина 8 символов без требования энтропии или блокировки частых паролей соответствует типичному учебному ТЗ, но слаба для высокопоставленных аккаунтов.
- **HTTP security headers.** В приложении нет программной установки HSTS, `X-Content-Type-Options`, CSP и других заголовков (частично это переносят на reverse proxy).
- **Файлы SQLite.** Риск связан с правами доступа ОС и копированием `app.db` при компромиссе узла это не ошибка приложения напрямую.
- **Передача пользовательских данных во внешний OpenRouter.** Данные и промпты уходят провайдеру нужна политика хранения и согласия в продукте.
- **Prompt injection.** Пользовательский контент управляет поведением модели традиционный класс LLM риска не RCE вашего приложения напрямую.

---

## 6. Положительные элементы архитектуры

- Хеш пароля через passlib/bcrypt без возврата пароля или хеша в ответах API.
- Унифицированная ошибка при неверном логине затрудняет перечисление несуществующих учёток по признаку логина против пароля при одном кейсе ошибки входа используется `UnauthorizedError` с общим текстом для обоих сценариев.
- Доступ к чату и истории через `user_id` из JWT снижает IDOR между пользователями на реализованных маршрутах.
- Запросы к БД через ORM снижают типичный риск SQL-инъекций в ручных запросах.
- Клиент OpenRouter формирует сообщения об ошибках провайдера без явного включения ключа API в ответ клиенту в просмотренном коде.

---

## 7. Ограничения ревью

Не проводился автоматический SAST/DAST, не анализировались бинарные артефакты, не проверялась инфраструктура развёртывания (TLS, WAF, логирование PII, SIEM). Рекомендуется перед продакшеном повторять оценку с учётом среды и угрозных моделей.
