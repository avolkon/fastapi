# Чеклист безопасности (повторяющиеся проверки)

Использовать перед релизом и после существенных изменений в `app` или инфраструктуре.

## Секреты и конфигурация

- [ ] `JWT_SECRET` в продакшене длинный случайный, не хранится в git, доступ ограничен.
- [ ] `JWT_ALG` только ожиданное значение (обычно `HS256`), нет смешения с нестандартными значениями в проде без обоснования.
- [ ] `OPENROUTER_API_KEY` и прочие секреты только в секрет-хранилище или защищённом окружении.
- [ ] `OPENROUTER_BASE_URL` в продакшене согласован с allowlist доверенных хостов (или зафиксирован константой для прод-сборки).

## Приложение и API

- [ ] Отзыв или сокращение жизни токенов согласованы с моделью угроз (refresh, блоклист при компромиссе).
- [ ] Включены лимиты запросов к `/auth/*` и к `POST /chat` (периметр: приложение или reverse proxy или API gateway).
- [ ] Политика размеров контекста LLM: верхний предел по сообщениям и по суммарному размеру согласован с бизнесом и биллингом.
- [ ] `/health` не раскрывает чувствительные метки среды или сведённые к минимуму по политики раскрытия.
- [ ] Авторизация по роли при появлении админских маршрутов сверена с источником прав в БД, а не только с полем JWT.
- [ ] Ответ об ошибках не содержит внутренних трасс без крайней необходимости и политики.

## Пересечение с браузером и клиентами

- [ ] `CORS_ORIGINS` содержит только нужные источники, без шаблонных `*` в продакшене.
- [ ] Решено, нужно ли `allow_credentials` при вашей схеме аутентификации Bearer-only.

## Поставка и зависимости

- [ ] Зафиксированы версии зависимостей (lock-file), выполнен аудит известных уязвимостей (`pip-audit`, Dependabot или аналог).
- [ ] План обновления `python-jose`, `cryptography`, `passlib`, `bcrypt` при выпусках security patches.

## Данные и LLM

- [ ] Принято решение по хранению истории чата персональных данных retention, экспорт.
- [ ] Документирован риск prompt injection для потребителей API и поддержки.

## Эксплуатация

- [ ] HTTPS на краю сервиса включён там, где токены проходят по сети.
- [ ] Права на файл SQLite ограничены УЗ процесса приложения резервное копирование не кладёт сырую БД в публичные хранилища без шифрования.

## Регламент после инцидентов

- [ ] Известная утечка `JWT_SECRET` влечёт смену секрета и инвалидацию всех старых access-токенов по возможности после внедрения механизма отзыва.
