# Верхнеуровневый план реализации защищённого API для LLM

**Объединённая версия:** синтез **исходной концепции** (дорожная карта по слоям и этапам) и **расширенной версии**, выровненной по `project/ТЗ_0.txt` и `project/Критерии_оценки_fastapi_llm.txt`, с блоком самопроверки перед сдачей.

**Навигация по разделам:** §0 путь входа · §1 архитектура · §2 стек · §3 компоненты и дерево ТЗ · §4 решения (JWT, LLM-контекст, ошибки, `create_app`) · §5 этапы · §6 критерии успеха · §7 самопроверка · §8 что усилено относительно черновика.

## 0. Структура каталогов и точка входа (сведение с ТЗ)

В **ТЗ** задан запуск вида:

`uv run uvicorn app.main:app --reload ...`

и дерево модулей с пакетом **`app/`** на верхнем уровне приложения (`app/core`, `app/db`, …). Критерии оценивают наличие каталогов **`app/`, `core/`, `db/`** и т.д. как часть приложения.

**Требование к реализации:** код разместить так, чтобы импорт `app.main` и команда из ТЗ сохраняли **смысл** при рабочем каталоге **`llm_p/`** (там `pyproject.toml` и пакет **`app/`**, см. **`llm_p/Структура_FastAPI_llm.txt`**). Команда `uv run uvicorn app.main:app ...` выполняется **из `llm_p/`**. В **`llm_p/README.md`** зафиксированы каталог запуска и установка через **`uv`** без ошибок импорта. Цель: проверяющий воспроизводит окружение только по документации.

### 0.1. Целевая структура модулей (по тексту ТЗ)

Имена файлов должны находиться «в своей зоне» и совпадать по смыслу с деревом из задания:

```text
app/
├── main.py                 # create_app(), /health, startup, роутеры
├── core/                   # config, security (JWT + bcrypt utils), errors
├── db/                     # Base, models, async session/engine
├── schemas/                # Pydantic: auth, user, chat
├── repositories/           # только ORM-доступ: users, chat_messages
├── services/               # openrouter_client (+ при необходимости __init__)
├── usecases/               # auth.py, chat.py
└── api/                    # deps.py, routes_auth.py, routes_chat.py
```

## 1. Архитектурная концепция

**Многослойная архитектура с чёткими границами ответственности:**

- **HTTP слой (`api`)** — маршрутизация, валидация входа (Pydantic), перевод доменных ошибок в HTTP (или общие handlers)
- **UseCase слой** — бизнес-логика, порядок шагов, оркестрация репозиториев и сервисов
- **Repository слой** — атомарные операции с БД (**только** SQLAlchemy / ORM)
- **Service слой** — внешние интеграции (**OpenRouter**), без знания о FastAPI-роутинге и без прямых обращений к БД

*Замечание:* в **`schemas`** нет отдельного «слоя исполнения» — это контракты API; они не смешиваются с репозиториями и не тянут `httpx`.

## 2. Технологический стек

- **Фреймворк:** FastAPI (async)
- **БД:** SQLite + **aiosqlite**
- **ORM:** SQLAlchemy 2.x (async)
- **Аутентификация:** JWT (**python-jose**) + bcrypt (**passlib**)
- **HTTP-клиент к LLM:** **httpx** (async)
- **Управление зависимостями:** **uv**
- **Линтер:** **ruff**

Пакеты и ограничения версий — как в блоке **`[project.dependencies]`** в **`ТЗ_0`** (включая `python-multipart` для форм логина в стиле OAuth2).

## 3. Ключевые компоненты системы

### 3.1. Инфраструктура

- **Settings** из **pydantic-settings** + `.env`: `APP_NAME`, `ENV`, JWT, путь SQLite, параметры OpenRouter (ключ, base URL, **модель**, referer/site URL и имя приложения для заголовков **`HTTP-Referer`** / **`X-Title`**)
- Async SQLite: строка **`sqlite+aiosqlite:///...`**
- **JWT / bcrypt** — в **`core/security`** (не БД)

### 3.2. Данные (SQLAlchemy)

- **`User`:** id, email, password_hash, **role**, created_at (уникальный индекс на email по ТЗ)
- **`ChatMessage`:** id, user_id (FK), role, content, created_at
- Связь **one-to-many**: пользователь ↔ сообщения

### 3.3. Репозитории

- **`UserRepository`:** поиск по email/id, создание (commit / refresh там, где заложено в ТЗ)
- **`ChatMessageRepository`:** вставка; выборка **последних N** с корректной сортировкой; **полное** удаление истории пользователя. Сборка списка `messages` для LLM — только в **`ChatUseCase`**.

### 3.4. Use cases

- **`AuthUseCase`:** регистрация с проверкой email, доменный **конфликт** при дубликате; логин — проверка пароля и выдача JWT (**payload по §4.1**); профиль по **user_id**, **не найден** при отсутствии строки в БД
- **`ChatUseCase`:** `system` (опционально) + история **`max_history`** + текущий **prompt**; вызов OpenRouter с **temperature**; сохранить в БД **user** и **assistant** сообщения; изоляция данных по пользователю

### 3.5. OpenRouter-клиент

- Класс (напр. **`OpenRouterClient`**): **`POST …/chat/completions`**, заголовки **Authorization**, **HTTP-Referer**, **X-Title** из настроек; тело: модель из конфигурации + **messages**.
- Отказ провайдера → доменная **`ExternalServiceError`** (или аналог), без «немого» падения процесса.

**Модель по умолчанию для балла в критериях:** `stepfun/step-3.5-flash:free` (переменная уровня `OPENROUTER_MODEL` или эквивалент).

### 3.6. API

- **`POST /auth/register`**, **`POST /auth/login`** (форма OAuth2 для Swagger — **`OAuth2PasswordRequestForm`**, `username` = email), **`GET /auth/me`**
- **`POST /chat`**, **`GET /chat/history`**, **`DELETE /chat/history`**
- **`GET /health`** по ТЗ
- При необходимости — **CORS** из конфига при сборке приложения

## 4. Ключевые технические решения

### 4.1. Аутентификация

- **`OAuth2PasswordBearer`** + кнопка **Authorize** в Swagger
- **JWT payload (по тексту ТЗ):** **`sub`** (идентификатор пользователя), **`role`**, **`exp`**, **`iat`**  
  (В черновой формуляции могли упоминаться только `sub/exp/iat` — для сдачи **обязательно добавить `role`**.)

### 4.2. Контекст для LLM

- Опциональная **`system`**-инструкция  
- **`max_history`** ограничивает число сохранённых сообщений из БД  
- **`temperature`** передаётся в запрос к провайдеру

### 4.3. Ошибки

Usecases и клиент выбрасывают **доменные** исключения; в API — **глобальные handlers** и/или тонкая обёртка в роутере → стабильные HTTP-коды.

| Замысел по ТЗ | Конвенция имени |
|---------------|----------------|
| конфликт (email занят) | `ConflictError` |
| не авторизован | `UnauthorizedError` |
| **запрет / нет прав** | **`ForbiddenError`** (+ HTTP 403; класс нужен тексту ТЗ, даже если в MVP почти не вызывается) |
| не найдено | `NotFoundError` |
| внешний сервис (OpenRouter) | `ExternalServiceError` |

Базовый класс доменной ошибки — в **`core/errors.py`**.

### 4.4. Инициализация

- **`create_app()`**, подключение роутеров; при **startup** — создание таблиц через **`Base.metadata.create_all`** совместно с **async engine** по сценарию из **`ТЗ_0`** (инфраструктурный код в `main`, без SQL и use case внутри роутеров).
- **`main`** без SQL и без бизнес-логики

## 7. Самопроверка при реализации и перед сдачей

Три опоры реализации: **границы слоёв**, **контролируемые сбои OpenRouter**, по желанию — **минимальные тесты** (критериями не формализованы, снижают регресс).

### 7.1. Чеклист слоёв

- **`api/routes_*`:** только Depends, схемы, вызов use case и/или общие handlers. **Не допускать:** Session/репозитории/`httpx`/SQL напрямую.  
- **`usecases/*`:** оркестрация; **только доменные** исключения. **Не:** `HTTPException`.  
- **`repositories/*`:** только persisted слой; **не** хешировать пароли и **не** вызывать OpenRouter.  
- **`services/openrouter_client.py`:** только HTTP/API OpenRouter → при ошибках **`ExternalServiceError`**. **Не:** обращаться к БД.  
- **`core/security`:** хеш и JWT (**с `role`**). **Не:** БД.  
- **`api/deps.py`:** сессии, сборка объектов; по ТЗ часть ошибок авторизации здесь можно отдавать как **401**.

### 7.2. Ошибки OpenRouter (минимум)

HTTP **4xx/5xx** от провайдера; **сеть/таймаут**; **200 без полезной нагрузки** (нет `choices` / контента) — везде **доменная** ошибка внешнего сервиса; в HTTP — один выбранный код (**напр. 502/503**) **без** «сырого traceback» клиенту.

### 7.3. Опциональные тесты

- Auth use case: дубликат email, неверный пароль  
- Chat use case: мок клиента LLM на успех (два сохранения в историю), на ошибку — зафиксировать согласованное поведение истории  
- Лёгкий интеграционный **`TestClient`** без реальной сети

Рационально подключать после прохождения ручного сценария **§5 (этап 7)**.
