ЭПИК 1. AUTH SERVICE — РЕГИСТРАЦИЯ, ЛОГИН, ВЫПУСК JWT

Статус: Реализовано (этапы 1-6)
Приоритет: Высокий
Цель: Создать независимый сервис аутентификации, который управляет пользователями и выдает JWT для Bot Service.

Границы ответственности:
1) Делает:
- хранит пользователей в БД;
- хеширует и проверяет пароли;
- выпускает JWT с полями sub, role, iat, exp;
- валидирует JWT для защищенных endpoint.

2) Не делает:
- не интегрируется с MAX API;
- не использует Redis для чат-состояния;
- не вызывает Bot Service или Celery.

Бизнес-результат эпика:
- Пользователь может зарегистрироваться.
- Пользователь может залогиниться и получить access token.
- По валидному токену можно получить профиль /auth/me.

Функциональный объем (MVP):
1) POST /auth/register
2) POST /auth/login
3) GET /auth/me
4) GET /health

Обязательные требования к качеству:
- Секреты только из .env;
- password_hash не возвращается наружу;
- ошибки нормализованы через единый слой исключений;
- lifespan вместо deprecated startup/shutdown событий;
- тесты модульные и интеграционные.

Артефакты по итогам эпика:
- реализованные модули auth_service (core/db/repository/usecase/api);
- конфиги окружения и pytest.ini;
- тестовый набор для позитивных и негативных сценариев;
- обновленная документация по запуску и проверке.

Definition of Done:
1) Все endpoint работают по контракту.
2) JWT корректно формируется и валидируется (exp, подпись, sub).
3) Негативные кейсы возвращают корректные статусы (401/409/404).
4) Автотесты зеленые.
5) Нет утечки секретов и чувствительных данных.
