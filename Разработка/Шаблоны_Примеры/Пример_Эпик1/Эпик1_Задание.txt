# 🎯 Задание на разработку: EPIC1-STAGE1 Каркас приложения и конфигурация

## Контекст
Этап 1 открывает реализацию Эпика 1 (Auth Service). На этом этапе фиксируется архитектурный каркас, чтобы все следующие этапы (БД, security, API, тесты) опирались на единые настройки и структуру модулей.

## Текущее состояние
**Файл:** `project/services/auth_service/app/main.py`, `project/services/auth_service/app/core/config.py`  
**Функция/класс:** сборка FastAPI приложения и класс Settings  
**Текущий вывод/поведение:** структура и конфигурация не финализированы под требования эпика  
**Необходимый вывод/поведение:** сервис стабильно стартует, читает `.env`, отдает `/health`, использует lifespan

## Требования ТЗ
Auth Service должен быть независимым сервисом аутентификации, с конфигурацией через переменные окружения и базовыми системными endpoint. Бизнес-логика не должна находиться в `main.py`.

## Задача
Подготовить каркас Auth Service и централизованную конфигурацию для последующих этапов реализации.

## Технические требования

### 1. Изменения в каркасе приложения и конфигурации
**Логика работы:**
1. Зафиксировать структуру модулей (`main/core/api/db/repositories/usecases/schemas/tests`).
2. Вынести обязательные параметры в Settings через `.env`.
3. Настроить `lifespan` и endpoint `/health`.

### 2. Формат вывода
```
GET /health -> 200
{
  "status": "ok"
}
```

**Где:**
- `status` - индикатор доступности сервиса.

### 3. Обработка ошибок
```python
# Ошибка конфигурации окружения
Поднимать валидационную ошибку Settings при отсутствии обязательных переменных.

# Ошибка старта приложения
Логировать причину и завершать запуск без частично инициализированного состояния.
```

## Файлы для изменения
- `project/services/auth_service/app/main.py` - сборка приложения, lifespan, /health.
- `project/services/auth_service/app/core/config.py` - settings через pydantic-settings.
- `project/services/auth_service/.env.example` - обязательные переменные окружения.
- **Не изменять:** `project/services/bot_service/*`.

# 🎯 Задание на разработку: EPIC1-STAGE3 Безопасность и JWT

## Контекст
Этап 3 формирует security-базу: хеширование паролей, выпуск JWT и валидация токенов.

## Текущее состояние
**Файл:** `project/services/auth_service/app/core/security.py`, `project/services/auth_service/app/core/exceptions.py`  
**Функция/класс:** security helpers и доменные исключения  
**Текущий вывод/поведение:** единый security-контракт не финализирован  
**Необходимый вывод/поведение:** стандартизированная и безопасная работа с паролями и токенами

## Требования ТЗ
JWT должен включать `sub`, `role`, `iat`, `exp`; подпись и срок действия обязаны валидироваться; ошибки должны быть доменными.

## Задача
Зафиксировать и реализовать единый security-контракт для auth-сценариев.

## Технические требования

### 1. Изменения в security модуле
**Логика работы:**
1. Определить алгоритм `hash_password` / `verify_password`.
2. Реализовать `create_access_token` с обязательными claims.
3. Реализовать `decode_token` с корректной классификацией ошибок.

### 2. Формат вывода
```
JWT payload:
{
  "sub": "<user_id>",
  "role": "<user|admin>",
  "iat": <timestamp>,
  "exp": <timestamp>
}
```

**Где:**
- `sub` - идентификатор пользователя.
- `role` - роль из доменной модели пользователя.

### 3. Обработка ошибок
```python
# Невалидная подпись токена
Поднимать InvalidTokenError.

# Истекший токен
Поднимать TokenExpiredError.
```

## Файлы для изменения
- `project/services/auth_service/app/core/security.py` - хеширование и JWT.
- `project/services/auth_service/app/core/exceptions.py` - доменные ошибки.
- **Не изменять:** repository и API контракты вне scope этапа.

## Пример реализации
```python
На этом этапе код не приводится. Реализация выполняется в отдельной итерации.
```

## Тестирование
не пишешь на этапе разработки

## Критерии приемки
- [ ] Password hashing реализован и согласован.
- [ ] JWT создается с полным набором claims.
- [ ] Ошибки токенов разделяются на invalid/expired.

## Примечания для разработчика
1. Не логировать JWT secret и токены целиком.
2. Использовать единый алгоритм из конфигурации.
3. Не смешивать инфраструктурные и доменные исключения.
