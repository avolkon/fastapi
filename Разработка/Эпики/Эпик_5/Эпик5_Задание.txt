# 🎯 Задание на разработку: EPIC5-TASK5.1 AuthUseCase

## Контекст
Сценарии регистрации, логина и профиля — ядро защиты API.

## Текущее состояние
**Файл:** `app/usecases/auth.py`  
**Функция/класс:** `AuthUseCase`  
**Текущий вывод/поведение:** логика в роутере или смешана с FastAPI  
**Необходимый вывод/поведение:** register/login/profile как в Arch_TechDesign; логин: единый `UnauthorizedError` для «нет user» и «плохой пароль»

## Требования ТЗ
ТЗ: usecase не использует HTTPException; регистрация с хешем; доменные ошибки.

## Задача
Реализовать AuthUseCase, использующий репозиторий и функции из `security.py`.

## Технические требования

### 1. Бизнес-логика auth
**Логика работы:**
1. `register`: проверка занятости email → `ConflictError`; хешируем пароль; создаём пользователя через репозиторий.
2. `login`: если user не найден **или** пароль неверный → **один** `UnauthorizedError` (без enumeration). При успехе — JWT через `create_access_token`.
3. `get_profile`: `NotFoundError` если пользователя нет.

### 2. Формат вывода
```
Выходные сущности через схемы API (эпик 6): UserPublic и TokenResponse на границе.
```

**Где:**
- JWT payload содержит `role` из пользователя.
- ответ register — публичные поля без пароля

### 3. Обработка ошибок
```python
# Репозиторий вернул integrity error
Отобразится из usecase как Conflict на уровне register (конкретная обработка — по реализации).
```

## Файлы для изменения
- `app/usecases/auth.py`
- `schemas` уточняются вместе с эпиком 6 при необходимости типов возврата.
- **Не изменять:** `api/routes*` в этой задаче не изменять или только типы импортов если нужно связать сборку объекта.


## Примечания для разработчика

## Время на выполнение

# 🎯 Задание на разработку: EPIC5-TASK5.2 ChatUseCase

## Контекст
Сценарий вопроса к LLM с сохранением пары сообщений пользователь/ассистент.

## Текущее состояние
**Файл:** `app/usecases/chat.py`  
**Функция/класс:** `ChatUseCase`  
**Текущий вывод/поведение:** нет сборки сообщений или неверный порядок сохранения  
**Необходимый вывод/поведение:** system (опционально) + история max_history + prompt; сохранить user сообщение → вызвать клиента → сохранить assistant ответ

## Требования ТЗ
ТЗ: порядок ask в usecase без SQL в роутере; сохранять и запрос пользователя и ответ модели.

## Задача
Реализовать методы ask/get_history/clear_history по tech design задаче 5.2.

## Технические требования

### 1. Бизнес-логика чата
**Логика работы:**
1. `ask`: взять до `max_history` сообщений репозитория; добавить текущий `prompt`; при наличии `system` включить первым сообщением;
2. сохранить user prompt;
3. `chat_completion`; сохранить ответ assistant;
4. вернуть ответ текстом клиенту (через схему ответа).
5. `get_history`/`clear_history`: делегировать репозиторию.

### 2. Формат вывода
```
Формат internal messages перед вызовом клиента список dict role/content.
```

**Где:**
- `temperature` и `max_history` из входного запроса API (этап 6).
- история изолирована по `user_id`.

### 3. Обработка ошибок
```python
# ExternalServiceError от клиента LLM
Пробросить без превращения в 500 необработанно — см. handlers эпика 6.
```
