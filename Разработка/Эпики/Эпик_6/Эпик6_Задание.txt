# 🎯 Задание на разработку: EPIC6-TASK6.1 Dependency Injection (`deps.py`)

## Контекст
Связать FastAPI с сессиями, репозиториями и usecase согласно слоям.

## Текущее состояние
**Файл:** `app/api/deps.py`  
**Функция/класс:** Depends-провайдеры, `oauth2_scheme`, `get_current_user_id`  
**Текущий вывод/поведение:** нет единых зависимостей  
**Необходимый вывод/поведение:** `get_db_session`; фабрики репозиториев/usecase; извлечение user_id из JWT; невалидный токен → HTTPException(401)

## Требования ТЗ
ТЗ: OAuth2PasswordBearer; get_current_user_id из sub; здесь можно 401 как инфраструктурная грань.

## Задача
Собрать DI-слой.

## Технические требования

### 1. FastAPI Depends
**Логика работы:**
1. Функции создания репозиториев/usecase объектов через `Depends`.
2. `get_current_user_id` читает bearer и декодирует через `decode_token`; при ошибке — 401.

### 2. Формат вывода
```
Depends-сигнатуры согласованы с роутерами задач 6.2–6.3.
```

**Где:**
- OpenRouter клиент создаётся как singleton/request scope по решению команды — задокументировать в коде README.

### 3. Обработка ошибок

## Требования ТЗ

## Задача

## Технические требования

### 1. HTTP Auth
**Логика работы:**
1. `POST /auth/register`: тело RegisterRequest.
2. `POST /auth/login`: form username=password flow из ТЗ/OpenAPI.
3. `GET /auth/me`: Bearer + профиль пользователя.
4. Обработчик или router-level перевод ошибок домена в коды из дизайна эпика 6.2.

### 2. Формат вывода
```
POST /auth/login возвращает access_token token_type bearer.
```

**Где:**
- token строка делегируется AuthUseCase.login.

### 3. Обработка ошибок
```python
# ConflictError
```

# 🎯 Задание на разработку: EPIC6-TASK6.4 Сборка приложения `create_app()`, `/health`, async DDL startup

## Контекст
Финальная склейка FastAPI приложения без бизнес-логики в `main`.

## Текущее состояние
**Файл:** `app/main.py` (+ при необходимости подключение роутеров)  
**Функция/класс:** `create_app`, lifespan, routers, exception handlers  
**Текущий вывод/поведение:** нет создания таблиц по async паттерну или нет `/health`  
**Необходимый вывод/поведение:** `create_app` подключает роутеры; lifespan с `conn.run_sync(Base.metadata.create_all)` (или паттерн ТЗ/SQLAlchemy async); handlers доменных ошибок; `/health`; опционально CORS

## Требования ТЗ
ТЗ: create_app без бизнес-логики; startup create_all асинхронно; `/health`; CORS если в конфиге.

## Задача
Реализовать инфраструктурную сборку приложения и старт цикла жизни БД.

## Технические требования

### 1. main слой приложения
**Логика работы:**
1. `create_app()` возвращает `FastAPI(...)`.
2. Подключить `routes_auth`, `routes_chat` с общим префиксом по ТЗ если нужно (`/auth` `/chat` уже внутри модулей).
3. `GET /health` JSON status + env.
4. lifespan: async DDL без синхронного bind к async engine (согласно Arch_TechDesign 6.4).
5. Глобальный handler для доменных `AppError` → HTTP коды консистентно с роутами.
6. Optional CORS из settings.

### 2. Формат вывода
```
GET /health -> {"status":"ok","env":... }
```

**Где:**
- значения из `settings.ENV`, `APP_NAME` при желании добавить вторым полем по договорённости не ломая ТЗ.

### 3. Обработка ошибок
```python
# Ошибки при DDL на старте
Лог + завершить подъём приложения чтобы не висеть полудохлым состоянием.
```

## Файлы для изменения
- `app/main.py`
- `app/api/__init__.py` только если нужно для экспорта router
- **Не изменять:** Не размещать use-case вызовы напрямую в `lifespan` кроме init БД если не требуется ТЗ
