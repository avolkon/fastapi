Построение защищённого API для работы с большой языковой моделью
Цель работы
Целью данной работы является разработка серверного приложения на FastAPI, предоставляющего защищённый API для взаимодействия с большой языковой моделью (LLM) через сервис OpenRouter. В рамках задания необходимо реализовать аутентификацию и авторизацию пользователей с использованием JWT, хранение данных в базе SQLite, а также корректно разделить ответственность между слоями приложения (API, бизнес-логика, доступ к данным).
Задание направлено на формирование практических навыков:
работы с FastAPI и асинхронным backend,
проектирования серверной архитектуры с разделением слоёв,
использования JWT для аутентификации,
интеграции внешних API (LLM),
работы с базой данных через SQLAlchemy,
управления зависимостями проекта через uv.
Установка и запуск проекта через uv
В этом проекте мы специально используем uv, потому что это современный и очень быстрый инструмент для работы с Python-проектами. Он ускоряет установку зависимостей, упрощает создание виртуального окружения и помогает держать проект в порядке через pyproject.toml. В учебных проектах это особенно полезно: меньше “магии”, меньше ручной рутины, и у студентов быстрее получается воспроизвести окружение на любом компьютере. По смыслу uv заменяет привычную связку python -m venv + pip install, но делает это заметно быстрее и аккуратнее.

      "aiosqlite>=0.20.0",
      "pydantic[email]>=2.7.0",
      "pydantic-settings>=2.3.0",
      "python-jose[cryptography]>=3.3.0",
      "passlib[bcrypt]>=1.7.4",

      "python-multipart>=0.0.9",

      "bcrypt==4.3.0",
      "ruff>=0.14.14",
    ]
    
    [tool.uv]
    dev-dependencies = []
    ` 

    uv pip install -r <(uv pip compile pyproject.toml)
     
После этого необходимо зарегистрироваться на платформе OpenRouter и получить API-ключ, который будет использоваться для взаимодействия с большой языковой моделью (LLM) через внешний сервис. Полученный API-ключ необходимо вставить в файл .env в строку OPENROUTER_API_KEY=, после знака равенства, без кавычек. Этот ключ будет использоваться серверным приложением для отправки запросов к OpenRouter.
Пример .env
APP_NAME=llm-p
ENV=local

JWT_SECRET=change_me_super_secret
JWT_ALG=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=60

SQLITE_PATH=./app.db

OPENROUTER_API_KEY=
OPENROUTER_BASE_URL=https://openrouter.ai/api/v1
OPENROUTER_MODEL=stepfun/step-3.5-flash:free
OPENROUTER_SITE_URL=https://example.com
OPENROUTER_APP_NAME=llm-fastapi-openrouter

app/main.py
В этом файле нужно собрать приложение FastAPI как единый объект и подключить всё, что относится к инфраструктуре приложения. Нужно создать функцию create_app(), внутри которой создаётся FastAPI(title=...), добавляется CORS (если он предусмотрен в конфиге), подключаются роутеры из app/api/routes_auth.py и app/api/routes_chat.py. Далее нужно обеспечить создание таблиц базы данных при старте приложения. Для этого на событии startup нужно открыть соединение с engine и выполнить Base.metadata.create_all. Также здесь нужно реализовать простой технический endpoint GET /health, который возвращает статус и окружение, чтобы можно было быстро проверять, что сервер запущен. В этом файле не должно быть бизнес-логики и работы с репозиториями. Здесь только сборка приложения и инфраструктура запуска.

app/core/config.py
В этом файле нужно реализовать конфигурацию приложения через pydantic-settings. Нужно создать класс Settings, который читает переменные окружения из файла .env, и объявить в нём поля конфигурации. Обязательные поля: название приложения, окружение, настройки JWT (секрет, алгоритм, время жизни access token), путь к SQLite, а также настройки OpenRouter (api key, base url, модель, referer, title). Затем нужно создать единственный экземпляр настроек settings = Settings(), который будет импортироваться в других частях проекта. Никакой логики, кроме чтения настроек, здесь быть не должно.

app/core/security.py
В этом файле нужно реализовать всю криптографию и безопасность на уровне “утилит”. Здесь должны быть функции для хеширования пароля и проверки пароля через passlib. Также здесь должна быть функция генерации JWT access token, которая формирует payload с sub (идентификатор пользователя), role (роль), exp (время истечения) и iat (время выдачи). Также должна быть функция декодирования токена, которая валидирует подпись и срок действия и возвращает payload. Этот файл не должен обращаться к базе данных и не должен знать, как устроены роуты. Он предоставляет только функции для использования в usecase и dependency-слое.

app/core/errors.py
В этом файле нужно описать собственные исключения приложения, которыми пользуются usecase и сервисы. Должны быть базовая ошибка приложения и типовые наследники: конфликт (например, email уже существует), неавторизован (неверный пароль), запрещено (нет прав), не найдено (объект в базе отсутствует), ошибка внешнего сервиса (например, OpenRouter вернул ошибку). Эти исключения нужны для того, чтобы бизнес-слой не зависел от FastAPI и не выбрасывал HTTPException. Usecase должен выбрасывать доменные ошибки, а роутер уже превращает их в HTTP-ответ.

app/db/base.py
В этом файле нужно объявить базовый класс SQLAlchemy для декларативных моделей. Должен быть класс Base(DeclarativeBase). Это общий базовый класс, от которого наследуются ORM-модели. Никакой дополнительной логики не требуется.

app/db/session.py
В этом файле нужно создать асинхронный engine SQLAlchemy и фабрику сессий. Нужно сформировать строку подключения к SQLite в формате sqlite+aiosqlite:///..., где путь берётся из настроек settings.sqlite_path. Далее нужно создать engine = create_async_engine(...) и AsyncSessionLocal = async_sessionmaker(...). Это инфраструктурный файл, он не должен содержать никаких запросов к таблицам, только настройку соединения.

app/db/models.py
В этом файле нужно описать ORM-модели. Нужно реализовать модель пользователя User с полями id, email, password_hash, role, created_at. Email должен быть уникальным и индексированным. Также нужно реализовать модель сообщения чата ChatMessage с полями id, user_id, role, content, created_at. user_id должен быть внешним ключом на таблицу пользователей. Также нужно настроить связи relationship между пользователем и сообщениями. Это файл описания схемы данных, он не должен содержать бизнес-логику и не должен содержать usecases.

app/schemas/auth.py
В этом файле нужно описать Pydantic-схемы для регистрации и токенов. Нужно создать модель RegisterRequest, где есть email и пароль, и обязательно задать ограничения на длину пароля. Нужно создать модель TokenResponse, где возвращается access_token и token_type="bearer". Если логин реализован в стиле OAuth2 для Swagger, отдельная схема LoginRequest может быть не нужна, потому что логин принимается через OAuth2PasswordRequestForm, но схему токена всё равно нужно иметь, чтобы документировать ответ.

app/schemas/user.py
В этом файле нужно описать публичную схему пользователя, которая используется в ответах. Нужно создать UserPublic с полями id, email, role и включить model_config = {"from_attributes": True}, чтобы FastAPI мог возвращать ORM-объекты напрямую как схему. Пароль и хеш пароля в публичную схему включать нельзя.

app/schemas/chat.py
В этом файле нужно описать схему запроса к чату и схему ответа. В ChatRequest нужно предусмотреть поле prompt как основной текст запроса. Нужно предусмотреть поле system как необязательную системную инструкцию. Нужно предусмотреть max_history, чтобы управлять тем, сколько сообщений брать из истории. Нужно предусмотреть temperature, чтобы студент понимал, как управлять “креативностью” модели. В ChatResponse нужно вернуть поле answer.

app/repositories/users.py
В этом файле нужно реализовать репозиторий пользователей. Репозиторий должен принимать AsyncSession в конструктор и хранить его как приватное поле. Нужно реализовать методы получения пользователя по email и по id. Нужно реализовать метод создания пользователя, который добавляет ORM-объект в сессию, делает commit и refresh. В этом файле нельзя хешировать пароль и нельзя создавать JWT. Репозиторий делает только операции доступа к данным.

app/repositories/chat_messages.py
В этом файле нужно реализовать репозиторий сообщений. Репозиторий должен уметь добавлять сообщение (сохранение роли и контента), получать последние N сообщений пользователя с сортировкой, а также удалять всю историю пользователя. В этом файле нельзя обращаться к OpenRouter и нельзя формировать логику “какие сообщения включать в контекст”. Репозиторий только читает и пишет данные.

app/services/openrouter_client.py
В этом файле нужно реализовать клиента для OpenRouter. Здесь должен быть класс клиента, который умеет делать запрос POST /chat/completions на базовый URL OpenRouter. Клиент должен собирать заголовки Authorization, HTTP-Referer и X-Title, которые берутся из настроек. Клиент должен отправлять JSON с моделью и списком messages в формате роли и контента. Если OpenRouter вернул код ошибки, клиент должен выбросить доменную ошибку внешнего сервиса. Этот файл не должен работать с базой данных и не должен знать про пользователей.

app/usecases/auth.py
В этом файле нужно реализовать бизнес-логику регистрации и логина. Usecase должен принимать репозиторий пользователей. В методе регистрации нужно проверить, что email не занят. Если занят, нужно выбросить доменную ошибку конфликта. Если свободен, нужно захешировать пароль и создать пользователя через репозиторий. В методе логина нужно найти пользователя по email, проверить пароль, и если всё корректно, выдать JWT через функцию из core/security.py. Если email не существует или пароль неверный, нужно выбросить доменную ошибку неавторизован. Также нужно реализовать метод получения профиля по user_id, который использует репозиторий и выбрасывает ошибку не найдено, если пользователя нет. Этот файл не должен содержать HTTPException и не должен зависеть от FastAPI.

app/usecases/chat.py
В этом файле нужно реализовать бизнес-логику общения с LLM. Usecase должен принимать репозиторий сообщений и клиент OpenRouter. В методе ask нужно собрать список messages для модели: при наличии system-инструкции добавить system-сообщение, затем получить из репозитория историю пользователя и добавить её в messages, затем добавить текущий prompt как user-сообщение. После этого нужно сохранить prompt в базе как сообщение роли user. Далее нужно вызвать OpenRouter и получить ответ. После получения ответа нужно сохранить его в базе как сообщение роли assistant и вернуть текст ответа. В этом файле должна быть логика формирования контекста и порядка действий. Этот файл не должен создавать сессии базы данных напрямую, он работает только через репозитории.

app/api/deps.py
В этом файле нужно реализовать dependency injection. Здесь должны быть функции, которые создают и предоставляют сессию базы данных, репозитории и usecase-объекты через Depends. Также здесь должна быть логика получения текущего пользователя по JWT. Для Swagger правильнее использовать OAuth2PasswordBearer и логин через OAuth2PasswordRequestForm, чтобы кнопка Authorize работала корректно. В get_current_user_id нужно декодировать токен и вернуть user_id из sub. В этом файле допускается перевод доменных ошибок аутентификации в HTTP 401, потому что это инфраструктурный слой API.

app/api/routes_auth.py
В этом файле нужно реализовать HTTP-эндпоинты авторизации. Эндпоинт регистрации должен принимать RegisterRequest, вызывать метод usecase регистрации и возвращать UserPublic. Эндпоинт логина должен работать в формате OAuth2 для Swagger, то есть принимать OAuth2PasswordRequestForm, использовать username как email и возвращать TokenResponse. Эндпоинт /auth/me должен получать user_id через dependency и возвращать профиль через usecase. В этом файле не должно быть кода SQLAlchemy, не должно быть commit/refresh, и не должно быть вызовов OpenRouter. Только вызовы usecase и преобразование доменных ошибок в HTTP-ответы.

app/api/routes_chat.py
В этом файле нужно реализовать HTTP-эндпоинты чата. Эндпоинт /chat должен принимать ChatRequest, получать user_id через dependency, вызывать usecase чата и возвращать ChatResponse. Эндпоинт /chat/history должен получать историю через usecase и возвращать список сообщений. Эндпоинт /chat/history с методом DELETE должен очищать историю. Этот файл тоже должен быть “тонким”: без SQL и без прямого вызова OpenRouter.
После завершения реализации проекта необходимо проверить качество и стиль кода с помощью линтера ruff и убедиться, что в проекте отсутствуют ошибки форматирования и нарушения правил оформления.

ruff check

Запуск приложения

uv run uvicorn app.main:app --reload --host 0.0.0.0 --port 8000 
После запуска приложения сслыке вы должны увидеть интерфейс Swagger, в котором доступна документация всех реализованных эндпоинтов и возможность их тестирования.

Далее необходимо пройти процедуру регистрации пользователя, после чего выполнить вход в систему и получить JWT-токен, который будет использоваться для доступа к защищённым эндпоинтам приложения.
После получения JWT-токена необходимо выполнить авторизацию в Swagger, нажав кнопку Authorize в правом верхнем углу интерфейса и вставив полученный токен.


Далее, после успешной авторизации, по эндпоинту POST /chat следует отправить запрос к LLM. Этот запрос будет обработан сервером, передан во внешний сервис OpenRouter, а полученный ответ возвращён клиенту и сохранён в истории диалога.

После получения ответа от языковой модели необходимо проверить, что история диалога корректно сохраняется в базе данных. Для этого следует обратиться к эндпоинту GET /chat/history и убедиться, что в ответе присутствуют ранее отправленные запросы пользователя и ответы модели.
